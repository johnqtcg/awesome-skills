# log-analyzer skill 解析

`log-analyzer` 是一套面向事件排查的日志分析框架。它的核心设计思想是：**高质量的日志分析，目标不是把过滤后的日志倒给你看，而是产出一份有证据支撑、可复现、可反驳的调查结论——先识别日志格式、脱敏、锁定时间窗口，再用统计判断信噪、用 `trace_id` 还原失败的请求链路，并且始终区分"最早出现的错误"和"真正的根因"，最后按严重程度分级，连同"这次没看到什么"一起交出去。** 因此它把格式识别、PII 脱敏、时间窗口、统计显著性、跨服务关联、因果纪律、数量上限七道门禁，连同 Lite / Standard / Strict 三种强度、按需加载参考文件、证据规则、反例目录和交接协议，串成一条固定流程。它定位很清楚：自己是事件响应的**上游**，产出的证据包直接交给 `incident-postmortem` 去写正式复盘。

## 1. 定义

`log-analyzer` 用于：

- 从日志里排查事件、分诊错误突增、抽取时间线
- 跨服务用 `trace_id` / `request_id` 关联，还原一次失败的请求
- 把信号和噪声区分开，不被正常波动的错误吓到
- 覆盖多种日志格式：纯文本、JSON（`slog` / `zap`）、syslog、journald、容器和 Kubernetes 日志
- 在需要写正式复盘时，把证据包交接给 `incident-postmortem`

它输出的不是一段过滤后的日志，而是一份结构固定的证据包：

- 分析强度（Lite / Standard / Strict）与选择理由
- 时间窗口、来源、覆盖是否完整、检测到的格式
- 执行摘要（先回答用户真正问的问题，并标注"已确认 / 假设 / 待佐证"各几条）
- 分级发现（每条含 ID、可信度、类别、位置、脱敏后的证据、推断、因果链、反驳条件、建议）
- 时间线、关联图、根因假设、建议
- 被抑制的项、执行状态、残余风险
- 交接协议（给 `incident-postmortem` 用的结构化字段）

从设计上看，它更像一个"日志调查治理框架"，而不是一个会帮你 `grep ERROR` 再把结果贴出来的提示词。它要回答的是"这个结论凭什么成立、还有哪些没看到"，而不只是"日志里有哪些报错"。

## 2. 背景与问题

这个 skill 要解决的，不是"模型不会用 `grep` 和 `jq`"，而是脱离框架的日志排查非常容易滑向几种"看起来在查、其实结论不可靠"的模式。

常见问题大致集中在 8 类：

| 问题 | 典型后果 |
|------|----------|
| 把最早的那条 ERROR 当成根因 | 真正的起因往往在第一条 ERROR 之前的 30 秒里（一次发布、一次配置推送、连接池告警），却被跳过 |
| 只数错误条数，不看分母 | 5000 万请求里 87 条错误被当成"突增"，其实是完全健康的背景噪声 |
| 引用日志时带出密钥 | 报告里原样贴出 `Bearer eyJ…`，等于又泄露了一次凭证 |
| 从不具代表性的切片下结论 | 拿一次 12 小时故障中间的 30 秒切片，就断言故障起止时间 |
| 罗列错误，不做关联 | 列了一千条报错，却没走一条完整的失败请求链路，说不清是哪一跳坏的 |
| 把症状当成起因 | 一个上游故障引发几百条下游超时，报告去数下游，指错了服务 |
| 悄悄丢弃发现 | 报了几条显眼的，剩下的既不报也不说，团队以为已经查全了 |
| 没查出问题就什么都不写 | "看起来没事"背后到底扫了哪些、覆盖全不全，无从判断 |

`log-analyzer` 的设计逻辑，就是先把"这是什么格式、时间窗口和覆盖到底如何、这个数放到分母里还显著吗、最早的错误是不是根因、失败的请求链路长什么样、有没有把密钥带出来、哪些结论只是假设"一一想清楚，再给出分级发现和根因假设。

## 3. 与常见替代方案的对比

| 维度 | `log-analyzer` skill | 直接 `grep` 日志 | 把日志整段丢给模型总结 |
|------|----------------------|------------------|------------------------|
| 最早错误 vs 根因 | 强（因果链纪律） | 无 | 弱（容易认第一条） |
| 信噪判断 | 强（必须有分母） | 无 | 弱 |
| 跨服务关联 | 强（走 `trace_id` 链路） | 很难 | 弱 |
| 级联归因 | 强（起因簇 vs 症状簇） | 无 | 弱 |
| 引用前脱敏 | 强制、硬门禁 | 无 | 通常忽略 |
| 覆盖 / 时间窗口诚实 | 强制说明 | 无 | 通常不提 |
| 结论可反驳 | 每条发现都带反驳条件 | 无 | 弱 |
| 可交接给复盘 | 有结构化交接协议 | 无 | 弱 |

它的价值不在"跑得比 `grep` 快"，而在把日志排查从"过滤 + 猜"，提升成一套有证据、可反驳、诚实交代边界、并且能直接交棒给复盘的调查流程。

## 4. 核心设计逻辑

### 4.1 先选分析强度，再动手

`log-analyzer` 在第 0 步就要求选定 `Lite / Standard / Strict`，默认 `Standard`。它明确拒绝一种误区：所有排查都用同一套重型流程。

分流规则是硬性的：单服务、窗口 ≤ 1 小时、量 < 100 MB 且没有安全 / 数据 / 客户影响信号，才走 `Lite`；一旦涉及 SEV-1/2、客户可见故障、疑似安全事件、数据损坏、3 个以上组件的关联，或者要交付复盘，就必须升到 `Strict`；其余走 `Standard`。发现数量的软上限也随强度变化（5 / 10 / 15）。这样简单的"看看现在是不是着火了"不会被写成一份重型报告，而真正的大事故会自动拿到调查级的深度。

### 4.2 七道门禁是硬阻断，跳过就是"契约违规"

这个 skill 最核心的架构，是把 SRE 的判断力拆成七道**串行硬门禁**：格式识别、PII 脱敏、时间窗口、统计显著性、跨服务关联、因果纪律、数量上限。任何一道没过，后续工作就停下，并在输出里明确报出来。

这层设计的意义在于：它不把"要不要做统计判断""要不要脱敏"交给临场感觉，而是变成必须逐一走过、且要在"执行状态"里自证的检查点。skill 甚至把"没加载对应参考文件就下结论"也算成契约违规——判断的依据必须是加载过的方法，而不是凭印象。

### 4.3 "最早出现的错误"不等于"根因"

这是整个 skill 的灵魂，对应 unit-test 里的 Defect-First。因果纪律门禁点出最常见的失败：`grep ERROR`、取最早那条、就宣布它是原因。它的硬规则是：**时间上最早的错误，不自动等于根因。**

它要求为每个主要假设构造一条因果链：症状 → 直接触发点 → 底层原因 → 放大因素，每一环都要有一条脱敏后的日志或一个指标撑着；链条从日志补不全时，只能写成"假设——待佐证：<来源>"，不能当成已确认。与之配套的是工作流里的"首次出现回溯"：对每一类高危错误，找到它在窗口里的**第一次**出现，并抓取它之前的 30 行上下文——因为原因通常住在那段"起跑"里，而不是第一条 ERROR 那一行。golden 用例 001 就把这条纪律锁死了：真正的起因是第一条 ERROR 前 30 秒的一次配置推送。

### 4.4 统计显著性：没有分母的计数没有意义

`log-analyzer` 在每一句"频繁""突增"上都要过统计门禁。核心一句话是：**一个孤立的计数没有意义，得有分母。** 它要求估算同窗口的请求 / 事件基数，尽量和等长的基线窗口比较，并且优先用比率（`errors/sec`、`error_ratio = errors / total`）而不是绝对数。

它给了几条可直接套用的经验锚点：`error_ratio` 持续 ≥ 1% 是信号；高流量服务里 < 0.01% 多半是背景噪声；同一类错误 ≥ 3 倍基线才算突增；100 万行里出现 1 次几乎肯定是噪声；但基线窗口里从没出现过的新错误类，无论多少次都要查。参考文档专门点了"百万分之一陷阱"：5000 万请求里 0.0002% 的错误，绝对数看着吓人，比率其实很健康。golden 用例 005 锁定的正是这一条——87 条错误要被判成噪声、甚至写成"无发现"，而不是"突增"。

### 4.5 跨服务关联：还原一条失败的请求，而不是罗列一千条错误

关联门禁的立场很鲜明：多服务排查里最有价值的一步，是把**一条**失败的请求端到端走完，而不是去总结一千条错误。做法是从第一条错误样本里抽出所有关联字段，在所有在范围内的日志源（服务、边车、网关、消息队列）里搜同一个 `trace_id` / `request_id`，按时间排序，逐跳标注服务、操作、状态、延迟和跨越的边界（HTTP / gRPC / Kafka / DB），最后汇成一张紧凑的关联图，而不是甩一堆原始 JSON。golden 用例 004 验证的就是这条端到端链路的走法。

一个很见功力的设计是：**如果日志里根本没有关联 ID，这本身就是一条高危可观测性发现**，因为没有它调试能力被根本性削弱。skill 还给了按成本排序的补救建议：网关注入 `request_id` → OpenTelemetry 自动埋点 → 结构化日志绑定上下文字段。golden 用例 008 锁定的正是这一条。

### 4.6 级联分析：区分"起因簇"和"症状簇"

真实故障里，日志往往被"症状簇"淹没——一个上游故障引发成百上千条下游错误。去数症状会放大严重度、还指错服务。`log-analyzer` 因此明确区分起因簇（出问题的那个组件自身的错误，通常很少，1–5 行，错误类别很具体，比如 `pool exhausted`、`deadlock`、`disk full`）和症状簇（依赖它的组件报的错，通常很多、错误类别趋同、数量还和流量成正比）。

它给的判据很实用：症状簇在时间上高度聚集、错误类别收敛到通用的超时 / 拒绝、数量随流量涨；而看错误活动的**头 30 秒**，起因几乎总在那里。报告规则是关键设计：只报**一条**起因发现加一段级联摘要，而不是并列 N 条。golden 用例 003 把它锁死——200 多条跨 5 个服务的 `context deadline exceeded` 必须收敛成一条数据库连接池耗尽的根因。用例 009 则是它的孪生场景：表面 12000 条错误，去重 `request_id` 只有 4000 个，说明是 3 倍的重试放大，要数去重后的请求数、并建议加熔断。

### 4.7 先脱敏再引用，且绝不事后补救

PII 脱敏是硬门禁，不是建议：报告**绝不能**把未脱敏的密钥原样回显给用户。skill 给了一张"必须脱敏"清单及其固定替换：Bearer / JWT → `Bearer ***REDACTED***`、各类 API key、URL 里的口令、邮箱、手机号、卡号、证件号、Cookie / session、私钥等。

两个程序上的关键决定：一是**先定脱敏集合、再引用，绝不写完报告再回头补**——因为那份未脱敏的草稿本身就是一次泄露（聊天记录、自动保存、屏幕共享都会留痕）；二是**永远不脱敏 `trace_id` / `request_id`**，它们不是秘密，却是分析的命脉。JSON 用字段级删改保留结构，文本用分层正则替换，之后再随机抽 20 行复查残留。真发现密钥已泄露，就按"已泄露必须轮换"处理、报一条高危安全发现，但**发现里也只写位置和类别，不再把密钥抄进去**。golden 用例 002 锁定的就是这条 Bearer 脱敏。

### 4.8 时间窗口要说清楚，覆盖不全就别下结论

每份报告都必须用绝对 UTC 写清分析窗口，以及这个窗口是从哪来的（文件、聚合器查询、`kubectl` / `journalctl` 参数），还要标 `Coverage: 完整 / 部分`——只要有日志轮转、保留策略、采样或分页可能截断了数据，就得标"部分"。如果用户只给了没有时间戳的片段，就直接写"窗口未知"，并停止一切基于时间的结论。golden 用例 007 验证的正是这条：拿到故障中段的 80 行切片，要标成覆盖不全、拒绝对起止时间下结论、通过"开放问题"去要更多数据。

这一条还牵出一个更隐蔽的坑——聚合器采样。Datadog 说"5000 条 ERROR"，若入库采样是 10%，真实量约 5 万条，所有比率都得按采样率放大。skill 要求把采样率写进"执行状态"，并把这类数字当成待核对项而不是确凿结论（golden 用例 010）。

### 4.9 证据规则与"三档可信度"：把认知谦逊写进输出

每一条发现都必须带四样东西：精确的来源位置、脱敏后的 1–5 行样本、由它得出的推断，以及**什么样的数据会推翻这个推断**（反驳条件）。可信度分三档明确标注：已确认 / 假设 / 假设——待佐证；把假设当成已确认，就是契约违规。

这层设计把"认知谦逊"变成了一等输出。skill 有一条明确取向：拿不准时往低了标可信度，而不是往高了标——因为一份没有证据支撑的自信报告，会把回滚、告警、复盘的行动项全部指向错误的方向。它甚至允许并要求"数据不足以下结论"成为一个正当的、必须写出来的结论。

### 4.10 数量上限 + 残余风险：绝不悄悄丢弃发现

发现有按强度的软上限（5 / 10 / 15），但超出上限的不会被默默删掉，而是转进"残余风险 / 调查缺口"。规则分三阶段：所有高危发现全部上报，绝不因为上限被砍；中危按"影响客户 → 阻塞排查 → 运维卫生"排序补位；低危有余位才报；被挤下去的转成一行摘要进残余风险，并在摘要里注明"另有 N 项转入残余风险"。

这跟 skill 里的"被抑制项"是一对：那些看着吓人、但被门禁判成噪声 / 健康重试 / 非用户可控的模式，要一行一行列出来并说明为什么压下。两者合起来保证：无论是"没报"还是"压下"，都有交代，不会有验证过的问题被静默丢弃。

### 4.11 没有发现，也要交代扫了什么

如果窗口内确实没有可操作的问题，skill 要求明确写"窗口内无可操作发现"，但仍然要给出分析强度、窗口与来源、一句执行摘要、执行状态、该窗口的 `error_ratio`、以及"有哪些是你没能看到的"。它有一条硬规则：**一份没有"执行状态"的"无发现"报告就是契约违规**——没查出问题，必须用"扫了什么"的证据来背书。这是这个设计对"可观测性表演"的回答：查无异常，也要和查出异常一样可审计。golden 用例 011 锁的就是这条。

### 4.12 参考文件按需加载，也是硬门禁

skill 带了 10 份参考文件（格式速查、关联、聚合器查询、统计方法、脱敏、级联分析、工具命令、反例、快速清单、示例输出），其中反例目录和快速清单**永远加载**，其余按场景触发。它明确规定：在关联、统计显著性、PII 处理上下结论，却没加载对应参考文件，就是契约违规。这保证方法论不是"记得多少算多少"，而是每次都从写下来的方法出发。

### 4.13 它是事件响应的上游，交棒给 incident-postmortem

`log-analyzer` 把自己的边界画得很窄，而且很清楚："写日志语句"归 `go-observability-review`，"设计告警 / 面板"归 `monitoring-alerting`，"写正式复盘"归 `incident-postmortem`，"没有日志证据的调试"归 `systematic-debugging`。

它的主交接对象是 `incident-postmortem`：产出一段结构化交接块（事件 ID、影响摘要、UTC 窗口、受影响服务、数据源、关键发现 ID、主假设、免责表述），这些字段直接喂进复盘 skill 的第一道门禁，让复盘跳过重复收集。其中"免责表述"必须用系统与流程的语言，绝不点名个人。它还会在合适时把工作分给 `monitoring-alerting`（发现缺告警时）或 `systematic-debugging`（日志不足以触底时）。golden 用例 012 验证的正是这套交接。这种"作为事件流水线里一环"的组合方式，和 `go-review-lead`、`stock-analysis-lead` 的多 skill 协作是同一种思路。

## 5. 这个设计解决了哪些具体问题

| 问题类型 | skill 中的对应设计 | 实际效果 |
|----------|-------------------|----------|
| 把最早的错误当根因 | 因果纪律门禁 + 首次出现回溯 | 抓到 ERROR 之前 30 秒里的真正起因 |
| 数错误不看分母 | 统计显著性门禁 | 不再把健康背景噪声当成突增 |
| 罗列错误、说不清哪坏了 | 关联门禁（走 `trace_id`） | 还原出一条完整的失败请求 |
| 把症状当起因、指错服务 | 级联分析（起因簇 vs 症状簇） | 只报一条根因 + 级联摘要 |
| 引用日志带出密钥 | PII 脱敏硬门禁 | 报告不再成为二次泄露面 |
| 从不代表性的切片下结论 | 时间窗口门禁 + 覆盖标注 | 覆盖不全时拒绝下起止结论 |
| 结论无法被检验 | 三档可信度 + 每条带反驳条件 | 回滚 / 告警 / 复盘不被错误自信误导 |
| 悄悄丢弃发现 | 数量上限 + 残余风险 + 被抑制项 | "没报"和"压下"都有交代 |
| 查无异常时无从判断 | 无发现也要给执行状态 | 查无异常也可审计 |

## 6. 主要亮点

### 6.1 它把日志分析从"过滤 + 猜"改造成"产出证据包"

整个 skill 存在的意义，就是拦住"`grep ERROR` → 倒出结果 → 猜原因"这条老路，产出的是结构化、可审计、可直接交接的证据包。

### 6.2 七道门禁把 SRE 判断力变成可自证的检查点

格式、脱敏、窗口、统计、关联、因果、数量七道串行门禁，跳过任何一道都算契约违规，并且要在"执行状态"里自证。判断不靠感觉，靠流程。

### 6.3 "最早错误 ≠ 根因" + 首次出现回溯，是很有辨识度的设计

因果链纪律加上"抓第一条 ERROR 前 30 行"，直击日志排查最常见的翻车点。

### 6.4 分母纪律让它不轻易"喊狼来了"

坚持先有分母再谈频繁，5000 万请求里的 87 条错误会被判成噪声，而不是事故。

### 6.5 认知谦逊被写进了输出

三档可信度、每条必带反驳条件、被抑制项、开放问题、残余风险，以及"数据不足"也算正当结论——这在排查类工具里非常少见。

### 6.6 脱敏是硬门禁，且先脱敏再引用

把报告本身当成潜在的泄露面，先定脱敏集合再引用、绝不事后补救，还专门规定不脱 `trace_id`。

### 6.7 它是事件流水线里的一环，而不是孤立工具

窄边界 + 明确的交接协议（主要交给 `incident-postmortem`），让它和复盘、告警、调试等 skill 组合成完整链路。

## 7. 什么时候适合用，什么时候不该硬用

| 场景 | 是否适合 | 原因 |
|------|----------|------|
| 从日志排查事件、分诊错误突增、抽时间线 | 非常适合 | 这是核心场景 |
| 多服务用 `trace_id` / `request_id` 关联失败请求 | 非常适合 | 关联门禁 + 级联分析很有针对性 |
| 判断"这波错误是信号还是噪声" | 非常适合 | 统计显著性门禁就是为此设计 |
| 复盘前准备证据包 | 非常适合 | 有直接喂给 `incident-postmortem` 的交接协议 |
| 在源码里写 / 改日志语句 | 不适合 | 交给 `go-observability-review` |
| 设计告警、面板、SLO | 不适合 | 交给 `monitoring-alerting` |
| 撰写正式的免责复盘文档 | 不适合 | 交给 `incident-postmortem`（本 skill 只产出证据） |
| 没有日志证据的代码调试 | 不适合 | 交给 `systematic-debugging` |

一个使用提醒：不要一上来就上 `Strict`。简单的"现在是不是着火了"用 `Lite` 就够，把重型的关联、级联、基线比较留给真正的大事故——强度和事件规模匹配，才是这个 skill 的用法。

## 8. 结论

`log-analyzer` 的真正亮点，不是它比 `grep` 快，而是它把日志排查里最容易走过场的部分固化了下来：先识别格式、脱敏、锁定窗口，再用分母判断信噪、用一条走通的请求链路代替一千条罗列，并且始终把"最早的错误"和"真正的根因"分开，最后按严重程度分级、连同没看到什么和每条结论的反驳条件一起交出去。

从设计上看，这个 skill 很清楚地体现了一条原则：**高质量的日志分析，关键不是把日志过滤得更干净，而是让每一条结论都能说清楚它的证据是什么、什么会推翻它、这个数放到分母里还成不成立、最早的错误到底是不是根因，以及这次分析有哪些根本没覆盖到。** 这也是它特别适合事件排查、跨服务关联和复盘前证据准备的原因——它交出的不是一段日志，而是一份能被下一步（无论是人还是 `incident-postmortem`）直接接手的调查结论。
