# Design: Deep Research Trusted Repository Evidence

## 1. Context

- **Problem / Purpose**: Repository evidence currently trusts structured JSON without proving that a commit, path, line, excerpt, subject, or test execution is real. Dirty working-tree matches are incorrectly labeled with `HEAD`, budgets reset on every command, report source ceilings are not enforced, and the automatic classifier misses common Chinese requests.
- **Who is affected**: Agents and reviewers relying on `deep-research` for auditable codebase or hybrid findings.
- **Success criteria**:
  - dirty or untracked code is emitted as `working-tree-unpinned`;
  - pinned code is re-read from a real Git commit and checked at the declared line;
  - commit existence and subject are revalidated;
  - runtime High confidence requires an explicitly replayed passing test;
  - retrieval and extraction usage persists across CLI invocations;
  - Sources contains only verified cited evidence and respects the mode ceiling;
  - the two reported Chinese classification examples select `codebase` and `deep`;
  - the CLI entry point is decomposed into responsibility-focused modules.

## 2. Constraints And Compatibility

- Keep the existing `python3 scripts/deep_research.py <subcommand>` entry point.
- Use only the Python standard library and existing tools (`git`, `rg`); add no dependency.
- Preserve Web evidence behavior and the canonical nine-section report.
- Treat evidence JSON as untrusted input. Validation must cross the filesystem, Git, or process boundary.
- Never execute a command from evidence implicitly. Test replay requires an explicit CLI flag and uses argument arrays with `shell=False`.
- Preserve the current dirty worktree; do not reset or overwrite unrelated changes.
- Non-goal: cryptographically protect local artifacts from a malicious user who controls both the filesystem and process. The design prevents declarative `status: passed` from becoming High without real replay.

## 3. Approaches Considered

The alternatives were generated by inversion (do not trust evidence records; reconstruct truth from authoritative boundaries) and decomposition (separate collection, verification, budgeting, reporting, and classification).

### Option A: Harden JSON schema only

Require more fields and hashes but do not re-read Git or rerun tests.

- Lowest migration cost.
- Still permits a complete fabricated record.
- Does not satisfy the stated audit boundary.

### Option B: Fixed Git snapshot only

Search only `git show` / `git grep` output and reject all working-tree or recorded test evidence.

- Strong pinned-code integrity.
- Simple validation model.
- Prevents legitimate current-worktree investigation and does not solve test execution or session budgets.

### Option C: Live boundary verification plus persistent session state

Search the working tree but pin only clean tracked files; validate referenced repository records against Git or the current filesystem; require explicit test replay for runtime High; persist budget use; render only verified cited sources; then split responsibilities into modules.

| Dimension (weight) | A: Schema only | B: Snapshot only | C: Boundary verification |
|---|---:|---:|---:|
| Evidence correctness (0.30) | 1 (0.30) | 4 (1.20) | 5 (1.50) |
| Auditability (0.20) | 2 (0.40) | 4 (0.80) | 5 (1.00) |
| Migration cost (0.15) | 5 (0.75) | 3 (0.45) | 2 (0.30) |
| Testing confidence (0.15) | 4 (0.60) | 4 (0.60) | 5 (0.75) |
| Maintainability (0.20) | 2 (0.40) | 3 (0.60) | 5 (1.00) |
| **Weighted total** | **2.45** | **3.65** | **4.55** |

## 4. Chosen Approach

Choose Option C in `Deep` mode. It is the only design that closes every reported trust boundary while keeping worktree research usable and preserving the current CLI surface.

## 5. Architecture / Components / Flow

### Repository collection

1. Resolve the Git top level and `HEAD`.
2. Search current files with `rg`.
3. For each match:
   - if the path is tracked, present in `HEAD`, and clean, record the real commit;
   - otherwise record `working-tree-unpinned`.
4. Store the repository root, relative path, line, excerpt, and provenance.

### Repository validation

For each referenced evidence ID:

- `code` with a commit: `git cat-file -e`, `git show <commit>:<path>`, then compare line and excerpt;
- unpinned `code`: resolve the path under the declared root and compare current line and excerpt;
- `commit`: prove the commit exists and compare its actual subject;
- `test`: reject declared status as proof; only a successful explicit replay verifies the record.

Invalid repository records remain unusable and add concrete issue codes.

### Trusted test execution

`run-test` executes an argv array with `shell=False`, bounded timeout, and a declared root. It records timestamps, duration, exit code, stdout/stderr summaries and hashes, and an execution ID.

Validation does not trust that receipt by itself for High confidence. `validate --replay-tests` or `report --replay-tests` explicitly reruns the recorded argv and derives the observed result.

### Session budget ledger

`plan --output <session.json>` initializes a versioned session artifact with cumulative usage and events.

- `retrieve --session` atomically reserves retrieval calls.
- `fetch-content --session` atomically reserves extraction attempts and processes only the remaining allowance.
- `report --session` derives its mode from the session and records the verified cited source count.
- Per-command parser ceilings remain defense-in-depth; the ledger is authoritative across invocations.

### Report source selection

Collect evidence keys from validated findings, analysis, consensus, and debate. Filter Web and repository artifacts to those keys, enforce `report_sources_max`, then create citation maps and Sources from that filtered set. Exceeding the ceiling blocks report generation instead of silently dropping citations.

### Classifier

Use explicit multilingual signal groups for repository, external, depth, comparison, security, and multi-provider requests. Explicit `--mode` and `--research-kind` remain authoritative.

### Module boundaries

- `planning.py`: multilingual classifier and mode budgets.
- `session.py`: versioned ledger and atomic reservations.
- `repository.py`: code search, Git validation, test execution/replay.
- `reporting.py`: verified cited-source selection and source ceilings.
- `deep_research.py`: compatibility CLI plus the existing Web retrieval,
  validation orchestration, and nine-section renderer.

This change deliberately extracts the new trust boundaries first. Further
separation of the legacy Web/validation/rendering engine is a maintainability
follow-up and does not alter evidence semantics.

## 6. Failure Handling And Operational Notes

- Missing Git, invalid root, nonexistent commit, path escape, missing blob, line mismatch, excerpt mismatch, subject mismatch, or failed replay yields a typed issue and unusable evidence.
- Unpinned worktree code may support Medium but never direct-code High.
- Test replay is opt-in because evidence files are untrusted command sources. The skill instructs the agent to inspect argv before enabling replay.
- Ledger writes use a lock and atomic replace. An over-budget reservation fails before external calls.
- A report exceeding the source ceiling is not written; the command explains how many verified sources exceed the selected mode.
- Local artifacts are auditable and boundary-verified, not cryptographically tamper-proof against a hostile filesystem owner.

## 7. Validation And Testing

- Reproduce the dirty-worktree false pin in a temporary Git repository.
- Reject nonexistent path, `deadbee`, line/excerpt mismatch, and commit-subject mismatch.
- Reject hand-written `status: passed`; verify a real `run-test` receipt only after explicit replay.
- Prove two cumulative reservations cannot exceed a session limit.
- Prove Sources omits unused inputs and blocks 9 Quick cited sources.
- Exercise the reported Chinese classifier inputs.
- Run all existing Web, contract, smoke, and golden tests after module extraction.
- Run skill validation, Python compilation, `git diff --check`, and MkDocs build.

## 8. Open Questions / Risks

- Test replay proves the command ran and passed, not that the command semantically covers the finding; Source Quality Notes must still expose the command.
- Git line validation is intentionally strict. Moving code between collection and validation invalidates the evidence and requires recollection.
- Concurrent session commands need file locking; the implementation is optimized for local macOS/Linux environments where `fcntl` exists.
